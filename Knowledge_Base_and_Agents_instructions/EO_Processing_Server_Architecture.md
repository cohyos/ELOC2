## Executive Summary

ELOC2 currently simulates EO sensor output as synthetic bearing measurements and classification labels generated by the `ScenarioRunner`. No actual image data is captured, processed, or analyzed. For a production deployment where real EO sensors (thermal imagers, daylight cameras) feed the C2 system, a dedicated **EO Processing Server** is required to bridge the gap between raw video frames and the structured `SourceObservation` format that the fusion pipeline consumes.

This document specifies the architecture of that server: a Python-based microservice that ingests live video streams, runs real-time object detection and classification, extracts angular bearing measurements from pixel coordinates using calibrated camera models, and emits observations into the ELOC2 fusion pipeline via gRPC or WebSocket.

The EO Processing Server is designed as a **sidecar to the existing C2 system**, not a replacement. It slots into the same data flow that the simulator currently occupies, producing identical `SourceObservation` records so that the TrackManager, correlator, fuser, and workstation require zero changes.

## Current State

In the demonstrator, EO data flows as follows:

```
ScenarioRunner.step()
  └─> generateEoBearing(target, sensor)
        └─> SimulationEvent { type: 'eo-bearing', bearing, sensorId, ... }
              └─> LiveEngine.processSimEvent()
                    └─> TrackManager.processObservation(sourceObs)
```

The `generateEoBearing` function computes a true bearing from sensor to target, adds Gaussian noise, and wraps the result as a `SourceObservation` with `sensorFrame: 'eo'`. There is no image data, no pixel-level processing, no learned model, and no real sensor interface.

Key limitations of the current simulation:

| Aspect              | Current (Simulated)           | Required (Production)              |
|:---------------------|:------------------------------|:-----------------------------------|
| Input                | Computed geometry             | Raw video frames (RTSP/GigE)       |
| Detection            | Known target list             | ML object detection                |
| Classification       | Random assignment             | CNN classifier on cropped ROI      |
| Bearing measurement  | True bearing + Gaussian noise | Sub-pixel centroid + camera model  |
| Latency              | Instant (same process)        | <100ms end-to-end                  |
| Throughput           | Unlimited (no real frames)    | 30 fps per sensor                  |
| Failure modes        | None                          | Occlusion, clutter, drift, fog     |

### Ingestion Layer

The ingestion layer manages connections to physical EO sensors. Each sensor gets a dedicated capture thread to avoid frame drops.

**Supported protocols:**

| Protocol     | Use Case                         | Library          |
|:-------------|:---------------------------------|:-----------------|
| RTSP         | IP cameras, thermal imagers      | OpenCV VideoCapture / GStreamer |
| GigE Vision  | Industrial/military cameras      | Aravis (Python bindings)       |
| USB/V4L2     | Development/lab cameras          | OpenCV VideoCapture            |
| File replay  | Testing with recorded footage    | OpenCV VideoCapture            |

Each capture thread writes frames into a bounded ring buffer (capacity: 4 frames). If the processing pipeline falls behind, older frames are dropped rather than queued — staleness is worse than loss in a real-time C2 context.

**Frame metadata** is attached at capture time:
- `sensor_id`: maps to ELOC2 `SensorId`
- `timestamp_utc`: NTP-synchronized capture time
- `sequence_number`: monotonic counter for drop detection
- `sensor_pose`: current pan/tilt/position (from gimbal controller)

### Processing Pipeline

The pipeline runs as a sequence of stages connected by in-process queues. Each stage is independently profiled.

**Stage 1 — Preprocessing**

- Resize to model input dimensions (640x640 for YOLO, configurable)
- Thermal normalization (AGC, histogram equalization) for IR sensors
- Optional temporal noise reduction (rolling average of 2-3 frames)
- Lens distortion correction using calibration parameters

**Stage 2 — Detection**

- Primary model: YOLOv8-m (medium) exported to ONNX format
- Inference via ONNX Runtime with CUDA Execution Provider
- NMS threshold: 0.45, confidence threshold: 0.3 (tunable per deployment)
- Output: list of bounding boxes with confidence scores
- Fallback: background subtraction (MOG2) for degraded conditions

**Stage 3 — Classification**

- Runs on cropped ROIs from detection stage
- Model: EfficientNet-B2 fine-tuned on aerial target classes
- Classes aligned with ELOC2 `TargetClassification`:
  - `fixed-wing`, `rotary-wing`, `uav`, `missile`, `unknown`
- Confidence threshold for positive classification: 0.7
- Below threshold, label is `unknown` with raw scores attached

**Stage 4 — Bearing Extraction**

- Compute sub-pixel centroid of detection bounding box
- Apply camera intrinsic matrix (focal length, principal point) to get angular offset
- Combine with gimbal pan/tilt and sensor geodetic position to get true azimuth/elevation
- Bearing uncertainty derived from bounding box size and pixel resolution
- Output: `BearingMeasurement { azimuth_deg, elevation_deg, sigma_az, sigma_el }`

**Stage 5 — Output Formatting**

- Map detection + classification + bearing into ELOC2 `SourceObservation`:
  - `sensorFrame: 'eo'`
  - `position`: bearing-derived position (range estimated or set to a default bearing-only value)
  - `covariance`: populated from bearing uncertainty and assumed range bracket
- Attach extended metadata (bounding box, classification scores, frame snapshot)
- Emit via gRPC stream or WebSocket message

### Output Interface

The server exposes two output interfaces:

**gRPC (primary, recommended for production):**

```protobuf
service EoProcessing {
  // Server-streaming: one stream per sensor
  rpc StreamObservations(SensorRequest) returns (stream EoObservation);

  // Health and status
  rpc GetStatus(Empty) returns (ServerStatus);

  // Camera calibration upload
  rpc SetCalibration(CalibrationData) returns (CalibrationResponse);
}

message EoObservation {
  string observation_id = 1;
  string sensor_id = 2;
  int64 timestamp_ms = 3;
  BearingMeasurement bearing = 4;
  BoundingBox bbox = 5;
  Classification classification = 6;
  bytes thumbnail_jpeg = 7;  // optional cropped ROI
}
```

**WebSocket (for development and demo use):**

- Connects to the ELOC2 API server at `ws://api:3001/ws`
- Sends `eo.observation` messages in the same JSON format as the simulator
- Enables drop-in replacement of simulated EO data with live data

## Integration with ELOC2

### SourceObservation Mapping

The EO Processing Server output maps directly to the existing domain type:

```
EoObservation (from server)        SourceObservation (ELOC2 domain)
─────────────────────────────      ──────────────────────────────────
observation_id                 --> observationId
sensor_id                      --> sensorId (branded SensorId)
timestamp_ms                   --> timestamp (branded Timestamp)
bearing -> projected position  --> position: Position3D
(not available for bearing-only) -> velocity: undefined
bearing uncertainty            --> covariance: Covariance3x3
'eo'                           --> sensorFrame: 'eo'
```

### Classification Enrichment

The server provides classification data that the current simulator generates randomly. In the ELOC2 pipeline, this feeds into:

- `EoManagementModule.classifyTarget()` — uses server classification instead of simulated
- `SystemTrack.classification` — updated via fusion
- `QualityAssessor` — classification confidence factors into quality scoring

### Bearing Extraction

Extracted bearings feed into the `@eloc2/geometry` triangulation pipeline identically to simulated bearings. When two or more EO sensors observe the same target, the existing `triangulate()` function produces a 3D fix. The bearing uncertainty from the camera model provides more realistic covariance than the fixed Gaussian noise in the simulator.

- CPU: 4 vCPUs (for preprocessing and I/O)
- RAM: 16 GB (frame buffers, model weights, batch queues)
- Network: 1 Gbps (for video ingestion; 4 sensors x 30fps x 1080p ~ 400 Mbps)
- Storage: 20 GB SSD (models, calibration data, log buffer)

## Deployment

### Kubernetes with GPU Node Pool

Recommended for production deployments with multiple sensors.

```yaml
# eo-processing-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: eo-processing
  namespace: eloc2
spec:
  replicas: 2  # one per sensor pair
  selector:
    matchLabels:
      app: eo-processing
  template:
    spec:
      nodeSelector:
        cloud.google.com/gke-accelerator: nvidia-tesla-t4
      containers:
      - name: eo-processor
        image: gcr.io/eloc2demo/eo-processing:latest
        resources:
          limits:
            nvidia.com/gpu: 1
            memory: 16Gi
          requests:
            cpu: "4"
            memory: 8Gi
        env:
        - name: SENSOR_IDS
          value: "eo-1,eo-2"
        - name: ELOC2_API_URL
          value: "http://eloc2-api:3001"
        ports:
        - containerPort: 50051  # gRPC
        - containerPort: 9090   # Prometheus metrics
        livenessProbe:
          grpc:
            port: 50051
          periodSeconds: 10
```

### Cloud Run with GPU

For lighter deployments or development. Cloud Run GPU support (preview as of 2025) allows single-container GPU workloads without managing Kubernetes.

```bash
gcloud run deploy eo-processing \
  --image gcr.io/eloc2demo/eo-processing:latest \
  --gpu 1 --gpu-type nvidia-l4 \
  --cpu 4 --memory 16Gi \
  --min-instances 1 --max-instances 4 \
  --set-env-vars ELOC2_API_URL=https://eloc2-api-xyz.run.app \
  --region me-west1 \
  --project eloc2demo
```

## Scalability

The scaling unit is **one pod per sensor pair**. Each pod runs detection on a dedicated GPU, handling 2 sensors at 30 fps each (60 inferences/sec, well within T4 capacity for YOLOv8-m).

```
                    ┌─────────────────────┐
                    │   Load Balancer      │
                    │   (K8s Service)      │
                    └──────┬──────────────┘
                           │
            ┌──────────────┼──────────────┐
            │              │              │
     ┌──────▼──────┐ ┌────▼────────┐ ┌───▼───────────┐
     │ Pod A       │ │ Pod B       │ │ Pod C          │
     │ GPU: T4     │ │ GPU: T4     │ │ GPU: T4        │
     │ Sensors:    │ │ Sensors:    │ │ Sensors:        │
     │  eo-1, eo-2 │ │  eo-3, eo-4 │ │  eo-5, eo-6    │
     └─────────────┘ └─────────────┘ └────────────────┘
```

Horizontal scaling is driven by sensor count, not target count. Each pod processes all targets visible to its assigned sensors. The C2 system handles cross-sensor correlation and fusion.

For deployments beyond 10 sensors, a **model serving layer** (Triton Inference Server) can decouple inference from capture, allowing GPU sharing across sensors with dynamic batching.

## Security Considerations

- Video streams carry sensitive surveillance data; all RTSP connections should use RTSPS (TLS)
- gRPC channels between the EO server and ELOC2 API must use mTLS in production
- Thumbnail/ROI images stored in observation metadata should be encrypted at rest
- Model weights are export-controlled in some jurisdictions; container images must be stored in private registries
- Network segmentation: EO processing pods should be in a dedicated subnet with no public ingress

## Cost Estimate and Phasing

### Phase 1 — Proof of Concept (4-6 weeks, $5K)
- Single sensor, file replay input
- YOLOv8 detection on pre-recorded thermal video
- Output to ELOC2 via WebSocket (replace simulator)
- Validate bearing accuracy against ground truth
- Run on a single GCP T4 instance

### Phase 2 — Live Integration (6-8 weeks, $15K)
- RTSP ingestion from 2 live sensors
- Classification model fine-tuned on target domain
- gRPC output interface
- Kubernetes deployment with GPU node pool
- Latency and throughput validation

### Phase 3 — Production Hardening (4-6 weeks, $10K)
- 4+ sensor support, horizontal scaling
- Prometheus/Grafana monitoring dashboard
- Automated calibration pipeline
- Graceful degradation (fallback to background subtraction)
- Edge deployment option (Jetson AGX)

### Ongoing operational cost (4 sensors, 24/7):
- Cloud: ~$2,100/month (2x L4 instances + networking)
- Edge: ~$200/month (power, maintenance) after hardware purchase

## Risk Register

| Risk                                     | Impact | Mitigation                                  |
|:-----------------------------------------|:-------|:--------------------------------------------|
| Detection model fails on thermal imagery | High   | Train on domain-specific thermal dataset     |
| GPU unavailable in target cloud region   | Medium | Support CPU fallback (lower fps)             |
| RTSP stream drops under load             | Medium | Ring buffer + reconnect logic                |
| Bearing drift from uncalibrated gimbal   | High   | Periodic auto-calibration against star field |
| Latency exceeds 100ms budget             | Medium | Profile per-stage, optimize bottleneck       |
| Export control on ML models              | Low    | Use open-source base models only             |
